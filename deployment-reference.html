2. Five Deployment Scenarios

Scenario 1 — Local dev (docker-compose.yml)

Builds images from source on your machine. All three services run in Docker on localhost.

cp .env.example .env
docker compose up -d --build
# open http://localhost
docker compose down        # stop
docker compose down -v     # stop + delete database volume

Scenario 2 — Single server (docker-compose.prod.yml)

Pulls pre-built images from GitHub Container Registry (GHCR). No build step on the server — GitHub Actions builds and pushes on every merge to main.

cp .env.example .env   # set GITHUB_USER, GITHUB_REPO, DB credentials
docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d

Scenario 3 — Docker Swarm (docker-compose.swarm.yml)

Multi-node deployment. Uses overlay networks so containers on different machines can communicate. DB is pinned to the manager node.

# manager node
docker swarm init
# worker nodes
docker swarm join --token <token> <manager-ip>:2377

# deploy
docker stack deploy -c docker-compose.swarm.yml app
docker stack ls
docker stack rm app    # tear down

Scenario 4 — AWS ECS / Fargate (terraform/)

Serverless containers on AWS. Terraform provisions VPC, ECS cluster, task definitions, ALB, and RDS.

cd terraform
terraform init
terraform apply
# outputs the ALB URL when done
terraform destroy      # stop charges

Scenario 5 — AWS EKS / Kubernetes (terraform-eks/ + k8s/)

Managed Kubernetes on AWS. Includes HPA (scales pods on CPU) and Cluster Autoscaler (scales EC2 nodes).

cd terraform-eks
terraform init
terraform apply

# from repo root after apply:
bash setup-eks.sh

# get the frontend URL:
kubectl get service frontend -n app

# tear down:
cd terraform-eks && terraform destroy

3. CI/CD Pipeline

GitHub Actions (.github/workflows/ci-cd.yml) runs on every push to dev or main:

  1. Test — lint with ruff, run pytest (no AWS needed)
  2. Build & push — builds frontend and backend images, pushes to GHCR as :latest
  3. Deploy (commented out) — uncomment either the ECS or EKS job depending on your environment

ECS auto-deploy (uncomment in ci-cd.yml)

aws ecs update-service \
  --cluster docker-compose-app-cluster \
  --service docker-compose-app-backend \
  --force-new-deployment

EKS auto-deploy (uncomment in ci-cd.yml)

aws eks update-kubeconfig --region us-east-1 --name docker-compose-app-cluster
kubectl rollout restart deployment/frontend -n app
kubectl rollout restart deployment/backend -n app
Note: update-kubeconfig writes the cluster connection details and credentials into ~/.kube/config so kubectl knows which cluster to talk to. A fresh CI runner has no kubeconfig, so this step runs every pipeline.

Internal networking

In ECS, services communicated via an internal ALB (long AWS hostname). In Kubernetes, services communicate via internal DNS — much simpler:

# From any pod in the cluster, the backend is reachable at:
backend.app.svc.cluster.local:8000

# From within the same namespace (app), just:
backend:8000

HPA (Horizontal Pod Autoscaler)

Scales the number of backend pods based on CPU usage.

  • Threshold: 70% of the pod's requested CPU (250m)
  • Min replicas: 1, Max replicas: 5
  • Requires metrics-server to be installed in the cluster
  • Scale-down has a 5-minute cooldown by default
kubectl get hpa -n app                          # see current CPU% and replica count
kubectl top pod -n app                          # see raw CPU/memory usage

Cluster Autoscaler

Scales the number of EC2 nodes. Triggered when pods are stuck in Pending because no node has enough resources.

  • Not CPU-threshold based — it watches for Pending pods
  • Configured via node group tags: k8s.io/cluster-autoscaler/enabled
  • Runs as a pod in the kube-system namespace

Deployment strategy: Recreate vs RollingUpdate

Our deployments use strategy: Recreate — kills the old pod before starting the new one. This avoids Out-Of-Memory errors on a single t3.small node that can't fit two pods simultaneously. In production with multiple nodes, you'd use RollingUpdate for zero downtime.

6. IAM Roles in EKS

EKS needs two IAM roles (ECS only needed one):

RoleAssumed byPurpose
eks-cluster-roleEKS control planeManage network interfaces, load balancers
eks-node-roleEC2 worker nodesJoin cluster, pull images from ECR, send logs

The node role also has a custom ClusterAutoscalerPolicy attached, allowing the Cluster Autoscaler pod to call AWS Auto Scaling APIs to add/remove nodes.

7. EKS Node Storage

Each EC2 worker node gets an EBS (Elastic Block Store) volume automatically. This is ephemeral storage for the node's OS and container layers — not for application data. Key points:

  • t3.small nodes get a default 20 GB EBS volume
  • The database uses RDS (managed), not a volume on the node
  • If a node is terminated and replaced (by Cluster Autoscaler), the EBS volume is also replaced — no data loss because app data lives in RDS
  • In ECS/Fargate: no EBS — Fargate containers run on AWS-managed compute with no persistent disk
  • In Swarm: containers run on EC2 instances; the DB volume (pg_data) lives on whichever node the DB container is on

10. RDS Snapshots — Persistence and Restoration

What happens on terraform destroy

Both ECS and EKS terraform configs have skip_final_snapshot = false, so AWS automatically saves a snapshot before deleting the RDS instance. The instance is deleted, but data survives in the snapshot.

Terraform folderSnapshot nameStatus
terraform/ (ECS)docker-compose-app-final-snapshotAlready exists (created 2026-04-24)
terraform-eks/ (EKS)docker-compose-app-eks-final-snapshotCreated on next terraform destroy
Important: Both folders originally used the same snapshot name, which would cause the EKS destroy to fail with a duplicate snapshot error. Fixed by renaming the EKS snapshot identifier to include -eks-.

Do ECS and EKS share the same database?

No. Each terraform folder creates its own independent RDS instance with its own endpoint. Data added via EKS does not appear in ECS and vice versa. They are two completely separate databases.

How to restore from snapshot after terraform apply

terraform apply always creates a fresh empty database by default. It does NOT automatically restore from the snapshot. To restore, add snapshot_identifier to the aws_db_instance resource in rds.tf before running apply:

resource "aws_db_instance" "postgres" {
  identifier        = "${var.project_name}-db"
  engine            = "postgres"
  engine_version    = "16"
  instance_class    = "db.t3.micro"
  allocated_storage = 20

  # add this line — terraform creates the new instance from this snapshot instead of empty
  snapshot_identifier = "docker-compose-app-eks-final-snapshot"

  # username/password still required even when restoring from snapshot
  username = var.db_username
  password = var.db_password
  ...
}

Once you run terraform apply with that line, the new RDS instance starts with all your previous data. The snapshot_identifier is only used at creation time — subsequent applies ignore it since the resource already exists. You can leave the line in or remove it after.

Alternative — AWS Console: RDS → Snapshots → select snapshot → Restore DB Instance → creates a new instance → copy the new endpoint → update k8s/secret.yaml and re-run setup-eks.sh.

16. EKS Deployment — Troubleshooting Log

All errors encountered during the initial EKS deployment, with root cause and fix.

Problem 1 — Backend could not reach RDS (connection timeout)

sqlalchemy.exc.OperationalError: connection to server at "...rds.amazonaws.com" failed: Connection timed out

Root cause: The RDS security group only allowed inbound traffic from the ECS task security group (by group ID). EKS managed node group EC2 instances are not ECS tasks — they use the node's own security group, which was not whitelisted.

Fix: Changed the RDS security group ingress rule from referencing the ECS SG by ID to allowing the entire VPC CIDR block (10.0.0.0/16). Any resource inside the VPC can reach RDS on port 5432.

# terraform-eks/security_groups.tf
ingress {
  from_port   = 5432
  to_port     = 5432
  protocol    = "tcp"
  cidr_blocks = ["10.0.0.0/16"]   # was: source_security_group_id = aws_security_group.ecs_tasks.id
}

Problem 2 — nginx returned 499 / tasks could not be added

2024/xx/xx [error] recv() failed (104: Connection reset by peer) while reading response header from upstream

Root cause: The frontend nginx config used $BACKEND_HOST as the proxy target. In ECS, the internal ALB handled port translation (80 → 8000). In Kubernetes there is no ALB between services — nginx must specify the port explicitly. The backend ClusterIP service listens on port 8000, but the env var was set to just backend.app.svc.cluster.local with no port, so nginx was proxying to port 80 which nothing was listening on.

Fix: Added :8000 to the BACKEND_HOST env var in the frontend deployment.

# k8s/frontend-deployment.yaml
env:
  - name: BACKEND_HOST
    value: backend.app.svc.cluster.local:8000   # was: backend.app.svc.cluster.local

Problem 3 — Rolling update stuck in Pending (Insufficient memory)

0/1 nodes are available: 1 Insufficient memory.
preemption: 0/1 nodes are available: 1 No preemption victims found for incoming pod.

Root cause: The default deployment strategy is RollingUpdate, which starts the new pod before killing the old one. On a single t3.small node (2GB RAM), running two 512Mi pods simultaneously exhausted available memory. The new pod could never start, so the rollout was stuck forever.

Fix: Changed both deployments to strategy: Recreate, which kills the old pod first and then starts the new one. Brief downtime, but no OOM.

# k8s/backend-deployment.yaml and frontend-deployment.yaml
spec:
  strategy:
    type: Recreate   # was: RollingUpdate (default)

Problem 4 — Cluster Autoscaler: MissingRegion

Failed to create AWS Manager: MissingRegion: could not find region configuration

Root cause: The Cluster Autoscaler pod needs to know which AWS region the cluster is in. It was not inferring it from metadata.

Fix: Added AWS_REGION env var to the Cluster Autoscaler deployment.

env:
  - name: AWS_REGION
    value: "us-east-1"

Problem 5 — Cluster Autoscaler: NoCredentialProviders

NoCredentialProviders: no valid providers in chain

Root cause: The Cluster Autoscaler pod needs AWS credentials to call Auto Scaling APIs. The pod had no IAM role attached (IRSA not configured), and no credentials in the environment.

Fix: Created a Kubernetes secret from the local AWS CLI credentials and mounted it as env vars in the Cluster Autoscaler pod. In production, IRSA (IAM Roles for Service Accounts) is the proper solution — it gives the pod an IAM role without storing credentials as secrets.

# create the secret (done in setup-eks.sh)
kubectl create secret generic aws-credentials \
  --namespace kube-system \
  --from-literal=access-key-id=$(aws configure get aws_access_key_id) \
  --from-literal=secret-access-key=$(aws configure get aws_secret_access_key)

# referenced in cluster-autoscaler.yaml
env:
  - name: AWS_ACCESS_KEY_ID
    valueFrom:
      secretKeyRef:
        name: aws-credentials
        key: access-key-id
  - name: AWS_SECRET_ACCESS_KEY
    valueFrom:
      secretKeyRef:
        name: aws-credentials
        key: secret-access-key

Problem 6 — Cluster Autoscaler: leader election lock (pod stuck)

Failed to acquire lease kube-system/cluster-autoscaler
Waiting for leader election...

Root cause: Kubernetes uses a lease object (a lock) so only one Cluster Autoscaler replica runs at a time. A previous crashed pod still held the lease. The new pod saw the lease was taken and kept waiting — but the holder was dead, so nobody would ever release it.

Fix: Deleted the old crashing pod manually. Kubernetes replaced it with a fresh pod that successfully acquired the expired lease.

kubectl delete pod <old-autoscaler-pod> -n kube-system

Problem 7 — HPA showing "cpu: unknown"

NAME          REFERENCE            TARGETS          MINPODS   MAXPODS   REPLICAS
backend-hpa   Deployment/backend   cpu: unknown/70%   1         5         1

Root cause: HPA reads CPU metrics from the Kubernetes Metrics Server. The Metrics Server was not installed — EKS does not include it by default.

Fix: Installed metrics-server via its official manifest.

kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

After ~60 seconds, HPA began showing real CPU percentages and could make scaling decisions.

12. Local Dev — Troubleshooting Log

Problem — Backend crashes immediately on startup

backend_1  | sqlalchemy.exc.OperationalError: could not connect to server: Connection refused

Root cause: Docker Compose starts all containers roughly in parallel. The backend tries to connect to PostgreSQL before the database process inside the db container is ready to accept connections. depends_on only waits for the container to start, not for the database to be ready.

Fix: Add restart: on-failure to the backend service so it retries automatically. A more robust fix is a healthcheck on the db service combined with depends_on: condition: service_healthy.

# docker-compose.yml
backend:
  restart: on-failure   # retries until db is ready

# or with healthcheck (more explicit):
db:
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U $$POSTGRES_USER"]
    interval: 5s
    retries: 5

backend:
  depends_on:
    db:
      condition: service_healthy

Problem — Port already in use

Error: Bind for 0.0.0.0:80 failed: port is already allocated

Root cause: Another process (another Docker container, nginx, IIS on Windows) is already listening on that port.

Fix: Find and stop the conflicting process, or change FRONTEND_PORT in your .env file to a free port (e.g. 8080).

# find what's using port 80 on Linux/Mac
sudo lsof -i :80
# on Windows
netstat -ano | findstr :80

Problem — Code change not reflected after docker compose up

Root cause: docker compose up -d without --build reuses the existing image. Your code change is on disk but the container is still running the old image.

Fix: Always pass --build when you want to pick up code changes.

docker compose up -d --build

Problem — "No space left on device" inside container

ERROR: failed to solve: no space left on device

Root cause: Docker's local disk (used for images, build cache, and volumes) is full. This is common after many builds without cleanup.

Fix: Prune unused Docker resources.

docker system prune -f          # removes stopped containers, dangling images, unused networks
docker system prune -af         # also removes all unused images (more aggressive)
docker volume prune -f          # removes unused volumes (careful — deletes data)

Problem — Frontend can't reach backend (network error in browser)

Root cause: The browser runs on your host machine, not inside Docker. If the frontend JavaScript tries to call http://backend:8000 directly, the browser doesn't know what backend means — that hostname only resolves inside the Docker network.

Fix: API calls from the browser must go to localhost (the host machine), not to container names. The nginx reverse proxy handles the translation: the browser calls localhost/api/..., nginx (inside the container) forwards it to backend:8000 using Docker's internal DNS.

Problem — .env file not found / variables are empty

WARNING: The POSTGRES_USER variable is not set. Defaulting to a blank string.

Root cause: The .env file doesn't exist, or it's named wrong (e.g. .env.example was never copied).

Fix:

cp .env.example .env
# then edit .env with your values

Problem — Volume data persists unexpectedly between runs

Root cause: docker compose down stops containers but does NOT delete named volumes. The pg_data volume (and all its database data) survives.

Fix: Use -v flag to also remove volumes. Only do this when you want a clean database.

docker compose down       # stops containers, keeps volumes
docker compose down -v    # stops containers AND deletes volumes (wipes database)

13. Single Server — Troubleshooting Log

Problem — Can't pull image from GHCR (401 Unauthorized)

Error response from daemon: unauthorized: unauthenticated request

Root cause: GitHub Container Registry requires authentication even for public packages depending on visibility settings. The server has never logged in to GHCR.

Fix: Log in with a GitHub Personal Access Token (PAT) that has read:packages scope.

echo <YOUR_PAT> | docker login ghcr.io -u <YOUR_GITHUB_USER> --password-stdin
docker compose -f docker-compose.prod.yml pull

Problem — docker compose pull doesn't update the image (still runs old version)

Root cause: docker compose pull downloads the latest image but the running container is still using the old one. Pulling doesn't restart containers.

Fix: After pulling, recreate the containers.

docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d   # recreates containers with new image

Problem — App is up but database is empty after server reboot

Root cause: The pg_data Docker volume survives normal container stops, but if Docker itself was reinstalled or the volume was deleted, data is gone. Also, if the db container starts without the volume being mounted, it initializes a fresh empty database.

Fix: Never run docker compose down -v on a production server unless you intend to wipe data. Confirm the volume exists before starting.

docker volume ls | grep pg_data

Problem — Port 80 blocked by server firewall

Root cause: Cloud providers (AWS EC2, DigitalOcean, etc.) have a security group / firewall in front of the VM. Even if Docker is listening on port 80, external traffic can't reach it if the firewall doesn't allow it.

Fix: In the cloud console, add an inbound rule allowing TCP port 80 (and 443 for HTTPS) from 0.0.0.0/0.

Problem — Container exits immediately with no logs

Root cause: A missing environment variable or misconfigured .env on the server causes the app to crash before it can log anything useful.

Fix: Check logs and verify all required variables are set.

docker compose -f docker-compose.prod.yml logs backend
docker compose -f docker-compose.prod.yml config   # prints resolved config with all vars substituted

14. Docker Swarm — Troubleshooting Log

Problem — Environment variables not substituted in stack deploy

invalid interpolation format for "image" option in service "frontend":
  "ghcr.io/${GITHUB_USER}/..."

Root cause: docker stack deploy does NOT automatically read a .env file the way docker compose up does. Variables in the stack file are not substituted.

Fix: Export the variables into your shell environment before deploying, or use env $(cat .env | xargs).

export $(cat .env | xargs)
docker stack deploy -c docker-compose.swarm.yml app

Problem — Worker node can't join the swarm

Error response from daemon: rpc error: code = Unavailable

Root cause: The firewall on the manager node is blocking the ports Swarm uses for cluster communication.

Fix: Open these ports on the manager node's firewall/security group:

TCP 2377   # cluster management (join tokens, leader election)
TCP/UDP 7946   # node-to-node communication
UDP 4789   # overlay network traffic (VXLAN)

Problem — Service image not found on worker nodes

No such image: ghcr.io/user/repo-backend:latest

Root cause: Swarm dispatches tasks to worker nodes. Workers pull images independently from the registry. If a worker has never logged in to GHCR, it can't pull the image.

Fix: Log in to GHCR on every node in the swarm, not just the manager.

# run this on each worker node
echo <PAT> | docker login ghcr.io -u <GITHUB_USER> --password-stdin

Problem — Database volume not accessible / data lost after service reschedule

Root cause: The pg_data volume is a local volume on whichever node the db container happens to run on. If Swarm reschedules the db container to a different node (e.g. after a node failure), the new node has no pg_data volume — it starts fresh.

Fix: Pin the db service to the manager node with a placement constraint. That's why our swarm file has:

deploy:
  placement:
    constraints:
      - node.role == manager   # db always runs on manager, volume is always there

For production, use a distributed storage solution (NFS, GlusterFS, or a cloud-managed database like RDS) instead of a local volume.

Problem — Overlay network: containers on different nodes can't communicate

Root cause: Regular Docker bridge networks are per-host. Containers on node A and node B can't see each other on a bridge network. Overlay networks use VXLAN to tunnel traffic across nodes — but only work in Swarm mode.

Fix: Ensure networks in the stack file use driver: overlay. The attachable: true flag is needed if you want to run standalone containers (not services) on the overlay network for debugging.

networks:
  frontend_net:
    driver: overlay
  backend_net:
    driver: overlay

Problem — docker stack deploy shows services running but app doesn't respond

Root cause: docker service ls shows 1/1 replicas but the task may be in a crash loop. The service shows as running because Swarm counts the desired state, not necessarily the healthy state.

Fix: Check the actual task status.

docker service ps app_frontend              # shows task history including failed attempts
docker service logs app_frontend            # aggregated logs from all replicas
docker service inspect app_frontend --pretty

Problem — Rollback goes to wrong version

Root cause: Swarm only keeps the previous task spec — one version back. If you deploy twice, the first version is gone from Swarm's memory.

Fix: Tag images with git commit SHAs in addition to :latest so you can manually roll back to any version by updating the image reference.

# in CI/CD, tag with both latest and the commit SHA
docker tag repo-backend:latest ghcr.io/user/repo-backend:$GITHUB_SHA
docker push ghcr.io/user/repo-backend:$GITHUB_SHA

# then to roll back to a specific commit:
docker service update --image ghcr.io/user/repo-backend:abc1234 app_backend

15. ECS Deployment — Troubleshooting Log

Common issues when deploying or updating the ECS/Fargate stack.

VSCode — Terraform comments not colored

Root cause: VSCode has no grammar tokens for .tf files without an extension. Comments appeared as plain white text.

Fix: Install the HashiCorp Terraform extension in VSCode. It adds syntax grammar for .tf files, enabling theme-based comment coloring.

ECS service not picking up new image after CI/CD push

Root cause: ECS task definitions pin to a specific image tag (or :latest). Pushing a new image to GHCR does not automatically restart the service — ECS only pulls a new image when a task is replaced.

Fix: Force a new deployment. ECS stops the running tasks and starts new ones, which pull the latest image.

aws ecs update-service \
  --cluster docker-compose-app-cluster \
  --service docker-compose-app-backend \
  --force-new-deployment

How ECS knows what to run (common confusion)

ECS does not use any docker-compose file. The task definition (defined in Terraform as aws_ecs_task_definition) specifies the image, CPU, memory, and environment variables. The service (aws_ecs_service) says how many copies to run and on which cluster. docker-compose.yml is only used locally.
